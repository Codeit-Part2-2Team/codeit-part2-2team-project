## 1. 프로젝트 맥락 및 목표

### 1.1 비즈니스 시나리오

가상 헬스케어 스타트업 **헬스잇(Health Eat)** 의 AI 엔지니어링팀 관점에서 프로젝트를 수행한다. 사용자는 모바일 앱으로 자신이 복용 중인 약을 촬영하고, 앱은 이미지 인식을 통해 해당 약의 이름과 정보를 제공한다. 이 위에서 복약 안전 정보, 약물 상호작용 경고, 건강 상태 연계 서비스 등이 구축될 수 있다.

### 1.2 기술적 요구사항

- **입력**: 사용자가 촬영한 이미지 (이미지당 최대 4개 알약 포함 가능)
- **출력**: 각 알약의 클래스(약품명)와 위치(바운딩 박스)
- **운영 환경 가정**: 모바일 단말 (On-device inference를 전제로 설계 방향 설정)

### 1.3 평가 구조

| 평가 단위 | 평가자 | 주요 근거 자료 |
|---|---|---|
| 과정 중 평가 (개인) | 멘토 | 멘토링·Discord 소통·협업 태도 |
| 결과 평가 (팀) | 주강사 | 발표, 보고서, GitHub repo |
| 결과 평가 (개인) | 주강사 | 협업일지, 발표 참여도 |

**Kaggle 점수는 평가 대상이 아님**. 선의의 경쟁 지표로만 활용하며, 점수 변동의 **원인 추적**이 본질적 학습 목표다.

### 1.4 팀 전체 공동 목표

본 프로젝트 가이드는 세 가지 목표를 제시한다. 팀도 이를 그대로 수용한다.

- 좋은 동료를 얻어가며 함께 성장하기

## 2. 기술 스택 결정: YOLOv26

### 2.1 모델 선택: **YOLOv26 단일 트랙**

**Baseline**: `yolo26n` (Nano) — 빠른 실험 사이클 및 On-device 배포 시나리오용
**최종 제출 후보**: `yolo26n` (On-device 스토리), `yolo26s` (Kaggle 점수 경쟁)

### 2.2 선택 근거

YOLOv26은 Ultralytics가 2026년 1월에 정식 릴리스한 최신 세대 실시간 객체 검출 모델이다. 다음 4개 축으로 평가한 결과 이번 프로젝트와의 적합도가 가장 높다.

| 평가 축 | YOLOv26 특성 | 프로젝트 적합도 |
|---|---|---|
| **정확도** | COCO 기준 동급 최고 수준 (Nano 변형 기준) | ○ |
| **엣지 배포** | NMS-free 설계, DFL 제거, CPU 최대 43% 고속화 | ◎ |
| **생태계 성숙도** | Ultralytics 공식 지원, 한국어 자료 다수 | ◎ |
| **익스포트** | ONNX/TensorRT/CoreML/TFLite 네이티브, INT8/FP16 양자화 지원 | ◎ |

**핵심 차별점**: YOLOv26은 "더 크고 더 정확한" 방향이 아니라 **"엣지 배포 단순화"** 방향으로 설계된 모델이다. 전통적 YOLO 파이프라인의 골칫거리였던 NMS 후처리와 DFL 모듈을 제거함으로써, 학습 후 모바일 배포까지의 경로를 크게 짧게 만들었다. 이는 본 프로젝트의 "모바일 앱에서 약품 인식" 시나리오와 정합한다.

### 2.3 대안 모델 비교 (보고서용 리서치)

| 모델 | 라이선스 | COCO mAP | 엣지 친화성 | 생태계 | 채택 여부 |
|---|---|---|---|---|---|
| **YOLOv26** | AGPL-3.0 | 동급 최고 (Nano) | ◎ | ◎ | **채택** |
| RF-DETR (Roboflow, 2025) | Apache 2.0 | 60.5 (최고) | △ (GPU 필수) | 중 | 미채택 |
| D-FINE (2024) | Apache 2.0 | 54~55 | ○ | 중 | 미채택 |
| YOLOX (Megvii) | Apache 2.0 | ~51 | ○ | 중 | 미채택 |

RF-DETR은 정확도 면에서 YOLOv26을 능가하지만 **CPU 추론이 1초/이미지 수준**으로 모바일 배포 시나리오와 맞지 않는다. D-FINE은 균형형이나 **문서·한국어 자료 부족**으로 3주 내 팀 학습 비용이 크다. 따라서 "가장 참신한 선택"이 아닌 "가장 일관된 서사를 만들 수 있는 선택"을 기준으로 YOLOv26을 채택한다.

### 2.4 라이선스 고려사항

YOLOv26은 Ultralytics의 정책에 따라 **AGPL-3.0 라이선스**로 배포된다. 이는 YOLOv5/v8/v11/v12를 포함한 전 Ultralytics YOLO 라인업에 공통 적용되는 라이선스다.

AGPL-3.0은 "파생 저작물의 소스 공개"를 의무화하며, 특히 네트워크 경유 서비스(SaaS·API 서버)도 공개 대상에 포함한다. 일반 상용 배포에는 상당한 제약이지만, **본 프로젝트에서는 실질적 블로커가 되지 않는다**.

**이유**: 프로젝트 가이드가 GitHub Repository 링크 제출을 필수로 명시하고 있어, 결과물은 어차피 오픈소스로 공개된다. AGPL의 소스 공개 조건이 자동으로 충족된다.

**보고서·발표 활용 포인트**: 이 라이선스 논의 자체를 **"비즈니스 관점 후속 의사결정 사항"** 섹션으로 보고서에 담으면 평가 항목 중 "결과를 비즈니스 관점에서 해석하고 시사점을 도출하였나요?"에 직접적으로 대응할 수 있다. 예시 문장:

> 본 모델의 상용 배포 시, Ultralytics Enterprise License 구매 또는 Apache 2.0 라이선스의 D-FINE 등 대안 모델로의 전환 검토가 필요함. 이는 헬스잇 제품화 단계의 비즈니스 의사결정 사항.

## 3. 협업 인프라: GitHub × Kaggle 역할 분리

### 3.1 핵심 원칙: Single Source of Truth

두 플랫폼을 병행 사용하되, **역할을 분명히 분리**한다.

| 구분 | GitHub | Kaggle |
|---|---|---|
| 역할 | **정본** (Source of Truth) | 실행 런타임 + 제출 전용 |
| 관리 대상 | 소스코드, 노트북, 보고서, 협업일지 인덱스 | Kernel (GitHub 미러), Submission |
| 버전관리 | Git (branch/PR/review) | Kernel 버전 이력 |
| 팀 협업 | PR 리뷰, Issue 트래킹, Projects 보드 | 팀 제출 계정 공유 |

**이유**: 평가 주체(주강사)가 GitHub repo를 평가하기 때문에 GitHub는 대체 불가하다. Kaggle은 무료 GPU와 제출 채널을 제공하므로 병행한다. 둘을 섞어 관리하면 반드시 동기화 지옥에 빠지므로 책임을 명확히 나눈다.

### 3.3 Git 브랜치 전략

**GitHub Flow 변형**을 채택한다. 3주 프로젝트에 Git Flow는 과하다.

- `main`: 항상 동작 가능한 상태 유지, 평가 대상 브랜치
- `develop`: 통합 브랜치 (선택 사용, 팀이 원하면 추가)
- `feat/<이름>-<기능>`: 개인 작업 브랜치 (예: `feat/seungjun-baseline-training`)
- `exp/<이름>-<실험내용>`: 실험 전용 브랜치 (예: `exp/dohyuk-mosaic-aug`)

**PR 규칙**:
- `main` 머지 전 **최소 1인 리뷰 필수**
- PR 제목 컨벤션: `[카테고리] 변경 요약` (예: `[data] COCO → YOLO 포맷 변환 스크립트 추가`)
- PR 본문에 변경 요약, 테스트 방법, 스크린샷(해당 시) 포함

**리뷰 원칙**: 거절보다 **함께 고치기** 모드를 지향한다. "이 줄 고쳐"가 아니라 "이 부분은 이렇게 바꾸면 어떨까?"로 제안형 코멘트를 사용한다. 코드 품질보다 **학습 효과와 팀 결속**이 이번 프로젝트에서 더 중요한 가치다.

### 3.5 Projects 보드 구성

GitHub Projects 보드 컬럼 구조 제안:

- **Backlog**: 아이디어·하고 싶은 작업
- **To Do (This Week)**: 이번 주 할 일
- **In Progress**: 진행 중
- **Review**: PR 리뷰 대기
- **Done**: 완료

각 카드에 담당자, 마감일, 라벨을 부여한다. 라벨 체계: `data`, `model`, `experiment`, `infra`, `docs`, `bug`, `priority-high`.

### 3.6 Kaggle 운영 규칙

- **팀 계정으로만 제출**. 개인 제출 금지 (가이드 명시 조항)

- **제출 스크립트 자동화**: `scripts/make_submission.py`로 추론 → 포맷 변환 → CSV 생성을 단일 명령으로 실행할 것. 수동 제출은 반드시 실수를 부른다.

## 4. 역할 분담 원칙

### 4.1 역할 정의 (기능 중심, 팀원 무관)

프로젝트 가이드가 제시한 4개 역할에 **보고서·비즈니스 해석** 축을 추가하여 5개로 재정의한다.

| 역할 | 핵심 책임 | 주요 산출물 |
|---|---|---|
| **Project Manager / Tech Lead** | 전체 일정·우선순위 관리, 아키텍처 의사결정, 코드 리뷰 품질 게이트 | 마일스톤, PR 리뷰, 기술 의사결정 기록 |
| **Model Engineer** | YOLOv26 학습·추론 파이프라인 구현, export 로직 | `src/models/`, 학습 스크립트 |
| **Data Engineer** | EDA, 어노테이션 포맷 변환, 전처리·증강, 데이터 파이프라인 | `src/data/`, 변환 스크립트, EDA 노트북 |
| **Experimentation Lead** | 실험 설계·실행, 하이퍼파라미터 탐색, 결과 로깅·비교 | 실험 로그, 결과 비교표, Kaggle 제출 |
| **Report & Analysis Lead** | 비즈니스 해석, 결과 시각화, 보고서 메인 작성, 발표 자료 주도 | 최종 보고서, 발표자료, 시각화 |

| 축 | 측정 방법 | 목표 수준 (조정 가능) |
|---|---|---|

- 전원이 최소 1건 이상 PR 머지
- 전원이 최소 1개 이상 실험 주도
- 협업일지 3주간 공백 없이 작성
- Daily standup 출석률 90% 이상

| 평가 항목 | 대응 산출물 |

| 결과 해석 및 비즈니스 시사점 | 보고서 §6 + 라이선스 섹션 |
